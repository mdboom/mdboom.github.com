Migrating to a Mac from Linux

Date Thu 08 October 2015 Tags mac / linux / emacs

I've used Linux as my primary driver 20 years now, and have arrived at a software development workflow I'm very happy with, based primarily around emacs, tmux, i3, zsh and [insert webbrowser here]. I need to write more about that some day, but in short, the things I like about it is:

  • The ability to do everything without leaving the keyboard.
  • The ability to customize, integrate and automate all of these tools so I'm not repeating the same rote tasks multiple times a day.
  • The ability to put all of the configuration in a Git repo so I can track changes to it over time and it's easy to set up new machines.

When I recently started at Continuum, I was provided with a shiny new MacBook Pro. While Continuum doesn't require I use it for my day-to-day development work, I thought I'd try to get it working as close as possible to what I'm used to.

Note

Some of this may sound like the rantings of a grumpy old man. It's not really about objectively better---well, ok, the poor integration between Unix and GUI apps sucks, let's just get that out of the way---but most of what's below is just about expectations and muscle memory. My goal here is just to make things easier for others making the switch and as a call for help for better solutions where things are still lacking. Please leave suggestions for ways I can do things better in the comments.

Software Packages

Macs don't have a built-in package manager (no, the App Store doesn't count), but there are a few third party ones: Fink, MacPorts, Nix, and homebrew (and Anaconda increasingly belongs on that list, though it remains less of a generalist). Homebrew seems to be the one with the most momentum behind it. It also has the ability to install native Mac apps (through Cask) which turns out to be very handy to set up software without a lot of clicking through websites and installers.

I won't reproduce installation instructions for Homebrew and Cask here, but it suffices to say that with these tools I can fully automate the installation of most of the software I needed from the command line, very much like a Linux package manager.

Terminal

Use iTerm2. It behaves more like Linux terminals out-of-the-box and is eminently configurable. It even supports middle-click-to-paste. Awesome.

Emacs

There are a bunch of ways to get Emacs on a Mac. In keeping with trying to do everything through Homebrew, I opted to install the Homebrew emacs with native Cocoa bindings, so I pass the --with-cocoa option to brew. Also note that HTTPS requests from Emacs seem to fail from emacs if I don't use the GNU TLS library, so I also pass --with-gnutls.

brew install emacs --with-cocoa --with-gnutls

You can then add this Emacs to the Applications menu using:

brew linkapps emacs

emacsclient

I use emacsclient a lot to easily open buffers from the command line (I have it aliased to e), and it's my default EDITOR for git and such. If you start an Emacs server from within Emacs using:

(edit-server-start)
(server-start)

then the emacsclient binary that Homebrew installed just works.

However, I also use emacsclient to automatically paste links from my web browser into org-mode task lists. This uses a special URL scheme, org-protocol to tell the operating system which application to use to open the link. On the web browser side, you set up a bookmarklet that passes the URL of the current page to the application handler for the custom URL. The bookmarklet looks like:

javascript:location.href='org-protocol://capture://'+encodeURIComponent(location.href)+'/'+encodeURIComponent(document.title)+'/'+encodeURIComponent(window.getSelection())

To get this to call emacs on Linux, it's simply a matter of telling your desktop environment which command to run for a particular custom URL scheme. On a Mac, you need to create a full Framework application to associate itself with the custom URL scheme, with a script inside it to delegate to the commandline emacsclient application.

Fortunately, I did find such a pre-made Framework app in EmacsClient.app. Unfortunately, it's designed for use with AquaMacs, not a Homebrew-built emacs. But that turns out to be fixable.

Download the EmacsClient.app zip file and expand it in your /Applications directory. Right-click on it and choose Show Package Contents, then navigate to Contents/Resources/Scripts/main.scpt. Open the script in Script Editor and replace the path to the AquaMacs emacsclient with the Homebrew one (which is normally /usr/local/bin/emacsclient). Save the script, and now you have something that associates the org-protocol:// URL scheme with emacsclient. Now it's one click to add a link to a webpage or webmail to your org-mode notes file.

Environment variables

By default, wmacs, or other native GUI applications, won't pick up any of the environment variables defined for the command line in files like ~/.profile, ~/.bashrc or ~/.zshrc, and friends. This means that the "inferior" applications that emacs launches may not match what you would expect in your commandline environment.

Searching for a solution to this problem isn't straightforward, since Apple has changed how environment variables work with seemingly every release of OS-X. The only reliable solution for Yosemite that I found is a set of scripts called osx-env-sync. To quote its author, Ersin Er:

On OS X, command line applications and GUI applications are treated differently. (Well this can be put in a more technically correct manner but this is what you experience from a user's point of view.) One fundamental difference is that although it's straightforward to feed command line applications with environment variables, it's not so for GUI applications. It's even harder to feed both types of applications from a single source of definitions. (Well, one particular workaround is launching GUI applications from command line.) Moreover, OS X's relevant means for setting up environment variables (or initializing programs in general) have been changing over time in consecutive releases which makes the situation worse. Hundreds of topics at Stack Overflow is the living proof of this mess.

Keyboard

Modifier keys

Native Mac apps tend to put common keyboard shortcuts on the ⌘ key, rather than the Ctrl key used on Windows and Linux. I tried to adjust to this for a few days, but ultimately, I decided I can't fight a lifetime on muscle memory on this one.

Thankfully, Mac OS-X makes it easy to adjust the modifier keys. Go to System Preferences -> Keyboard -> Keyboard -> Modifier Keys. For Emacs, add the following to your ~/.emacs.d/init.el file:

(setq mac-command-modifier 'control)

  • Letting someone else use my Mac (which happens occasionally when giving team presentations) is likely to make things explode. Note to self: don't share.
  • Other Unix or X11 apps that don't have the ability to remap keys may be a problem. I haven't run into any of these that I use yet, but it's something to be aware of.

Other Emacs keys

Emacs also has some other keys that are mapped differently by default on Macs.

The Home and End keys move to the beginning and end of the buffer, respectively, to match the behavior in native Mac apps. Let's change them to move to the beginning and end of the line like they do on Linux (a much more common operation):

(define-key global-map [home] 'beginning-of-line)
(define-key global-map [end] 'end-of-line)

The Menu key (present on many Windows keyboards, but not on the built-in Mac keyboard) emits C-p, so it moves the cursor up one line. I never use C-p in Emacs, so I can just remap it to do what Menu does on Linux:

(global-set-key (kbd "C-p") 'smex)

If you wanted to keep C-p working in Emacs, you'll probably need to use a third-party tool such as Seil to remap Menu to an unused key and then make that work in Emacs.

Tiling window manager

Perhaps the single most important productivity tool I use on Linux was a tiling window manager. It's very freeing to always be able to move between applications without leaving the keyboard, and to never worry about carefully rearranging windows so they aren't partially obscured. I've used various tiling window managers over the years, but the one that ultimately best fit my brain and I've really stuck with is i3.

The built-in window management tools on the Mac always leave me reaching for the Dramamine, what with all whizzing around and zooming in and out. There are a few alternative window management "tools" for the Mac. I ended up trying most of the free or free-to-try ones, most of them only handle the window placement part of the problem. Only one, Slate also supports moving focus with directional keys.

It's a little tricky to set up as the default keybindings aren't very intuitive, but it's very customizable and digging into its config file is well worth the effort. Even still, the end result is a far cry from i3. Windows aren't automatically arranged: you have to manually place each one as they open. There's no tabbed stacked windows, so when windows are stacked you have to flip through each of them in order to find the one you want (or fall back on ⌘-Tab). And when the screen locks, everything becomes misplaced and you have to rearrange the windows again, though that seems to be a problem in general on a Mac.

Tiling managers are a polarizing technology. As one who is in the "love it" camp, I feel that may be the one pain point that sends me running back.
